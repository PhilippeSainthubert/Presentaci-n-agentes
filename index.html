Arquitectura · Configuración de agentes

Así se configura un agente de IA por dentro.

Un agente no es magia ni una app que se instala: es un modelo que ejecuta instrucciones escritas, con acceso a herramientas acotadas, dentro de un loop supervisado. Este documento muestra exactamente cómo se define, qué puede tocar, qué frenos tiene y sobre qué stack corre.

componente 1 Modelo El motor de razonamiento (LLM). Se elige por tarea: más capacidad para análisis, más velocidad para tareas repetitivas.
+ componente 2 Instrucciones El rol, las reglas y los límites, escritos en un archivo versionado que se puede leer, auditar y aprobar.
+ componente 3 Herramientas Conectores (MCP/APIs) con permisos explícitos: qué puede leer, qué puede escribir, qué tiene prohibido.
+ componente 4 Loop El ciclo razonar → actuar → observar, con puntos de aprobación humana donde hay gasto o publicación.
01

La definición: el agente es un archivo

Cada agente se define en un archivo de texto versionado en Git — mismo formato del estándar abierto Agent Skills que hoy leen Claude, OpenAI, Microsoft y 30+ herramientas. Esto significa: cero caja negra (cualquiera puede leer qué hace), historial completo (cada cambio queda registrado) y portabilidad (no depende de un proveedor). Ejemplo real de un agente analista de Meta Ads:

agents/meta-ads-analyst.md 
# ── Identidad y alcance ──────────────────
name: meta-ads-analyst
description: Analiza performance diaria de
  campañas Meta y propone optimizaciones.
model: claude-sonnet # costo/velocidad por tarea

# ── Herramientas permitidas (allowlist) ──
tools:
  - mcp:meta-ads:reporting      # lectura
  - mcp:meta-ads:campaigns:read
  - mcp:slack:post:#ads-cliente  # avisos

# ── Límites duros ────────────────────────
budget_tokens_month: 2_000_000
requires_approval:
  - cualquier escritura en campañas
  - cambios de presupuesto
never:
  - crear/pausar campañas sin aprobación
  - tocar audiencias de otros clientes

## Rol
Sos el analista de Meta Ads del cliente.
Cada mañana (07:00) leés las métricas de
las últimas 24h vía el conector oficial.

## Reglas
1. Si el CPA sube >25% vs media 7d,
   alertá en Slack con diagnóstico.
2. Proponé variantes de creatividad para
   los 3 ads de peor rendimiento.
3. Toda propuesta de cambio se entrega
   como borrador — nunca la ejecutás vos.
frontmatter → identidad

Qué es y qué modelo usa

Nombre, propósito y modelo asignado. Un agente de análisis profundo usa un modelo más capaz; uno de tareas repetitivas, uno más rápido y barato.

tools → allowlist

Solo lo que está en la lista existe

El agente no "descubre" herramientas: si un conector no está declarado, no puede usarlo. Acá este agente lee reporting de Meta y postea en un canal de Slack. Nada más.

requires_approval / never

Frenos declarados, no prometidos

Las acciones con gasto real exigen aprobación humana y las prohibiciones son explícitas. Esto se aplica a nivel de sistema (hooks), no depende de que el modelo "se porte bien".

cuerpo → system prompt

Las instrucciones operativas

El rol y las reglas de negocio en lenguaje natural. Es la parte que se itera semana a semana con lo aprendido en operación.

02

La capa de herramientas: MCP y permisos

Los agentes se conectan a sistemas externos vía MCP (Model Context Protocol), el estándar abierto de conectores que adoptaron Anthropic, OpenAI, Google y las plataformas. Cada conexión tiene credencial propia con mínimo privilegio. Así queda el mapa de herramientas de los cuatro agentes tipo:

AgenteConectoresLectura (autónoma)Escritura (con gate)
QA de entregables playwright (browser headless sobre propiedades propias), lighthouse, visión multimodal del modelo autónoma crawl de landings, screenshots, validación de links/UTMs/schema — solo emite reportes; no escribe en ningún sistema
Contenido SEO mcp:dataforseo (keywords/SERP), mcp:search-console, mcp:cms autónoma research, métricas, contenido existente aprobación publicar en CMS — siempre tras edición humana
Meta Ads mcp:meta-ads — el conector oficial de Meta (OAuth directo, sin app de developer) autónoma reporting, insights, diagnóstico de señal Pixel/CAPI aprobación crear/editar campañas y presupuestos · nunca browser automation
LinkedIn Ads Marketing API oficial (tier por aprobación de LinkedIn) autónoma performance de campañas en cuentas administradas aprobación campañas preparadas, publica un humano · nunca automatizar el sitio

La distinción lectura autónoma / escritura con gate es la decisión de diseño central: un agente puede analizar 24/7 sin riesgo, pero toda acción con gasto o publicación pasa por un punto de aprobación implementado a nivel de sistema.

03

Guardrails: los frenos son código, no promesas

La lección de la industria en 2026 (tras los incidentes de seguridad de los agentes "always-on" mal aislados): los límites tienen que estar fuera del modelo, en la infraestructura. Seis mecanismos concretos:

/01

Hooks de pre-ejecución

Antes de cada acción del agente, un interceptor valida la llamada contra las reglas. Si la acción no está permitida, se bloquea — el modelo ni siquiera puede intentarlo.

PreToolUse → allow | deny | ask_human
/02

Credenciales por agente

Cada agente tiene su propia identidad y tokens con mínimo privilegio. Si un agente se compromete, el radio de daño es solo lo que ese agente podía tocar.

1 agente = 1 credencial = 1 scope
/03

Aislamiento de runtime

Los agentes always-on corren en entorno aislado (VM/sandbox), nunca en una máquina personal, y el gateway no se expone a internet.

sandbox + no-public-gateway
/04

Presupuesto con corte

Límite de consumo (tokens/costo) por agente y por mes. Warning al 80%, pausa automática al 100%. Un agente en loop no puede quemar presupuesto sin que nadie lo vea.

budget: warn@80% · pause@100%
/05

Approval gates

Las acciones marcadas como sensibles (gasto, publicación, datos de cliente) generan una solicitud de aprobación que un humano resuelve desde el panel — con contexto completo.

requires_approval → cola en panel
/06

Audit log inmutable

Cada tool call queda registrada: qué hizo el agente, cuándo, con qué input, qué costó. El historial no se puede editar — es la base de la confianza y del debugging.

append-only · por agente · por acción
04

El stack completo, en capas

La definición del agente (sección 01) es solo una capa. Para operar en serio hacen falta cuatro, cada una con un trabajo distinto:

Capa 4Plano de control

Qué hace

El tablero de operación: estado de cada agente, cola de aprobaciones, consumo/costos en tiempo real, asignación de tareas, alertas.

Implementación

Dashboard tipo Mission Control sobre el gateway de agentes. Es la única capa que ve el cliente además de los outputs.

Capa 3Harness / runtime

Qué hace

Ejecuta el loop del agente: sesiones de trabajo supervisadas, tareas programadas (crons) y agentes persistentes always-on.

Implementación

Claude Code / Agent SDK para operación supervisada; runtime always-on aislado para agentes persistentes con scheduling.

Capa 2Conocimiento

Qué hace

Lo que el agente sabe del negocio: contexto del cliente, brand, playbooks, decisiones, aprendizajes de operación. Sin esto, el agente es genérico.

Implementación

Repositorio Git versionado ("brain"): definiciones de agentes, skills compartidas y doctrina, con reglas de escritura y review.

Capa 1Modelo + conectores

Qué hace

El razonamiento (LLM) y los brazos (MCP/APIs oficiales de Meta, LinkedIn, CMS, analytics).

Implementación

Modelos según tarea (capacidad vs costo) + conectores estándar MCP con credenciales de mínimo privilegio.

05

Ciclo de vida: cómo se gana la autonomía

Un agente no nace autónomo — se gradúa. Cada etapa amplía permisos solo si la anterior funcionó, con evidencia del audit log:

Etapa 1

Definición

Se escribe el archivo del agente: rol, tools, límites, reglas. Review y aprobación por escrito antes del primer run.

Etapa 2

Operación supervisada

El agente corre en sesiones con humano presente. Solo lectura y borradores. Se itera el prompt con los errores reales.

Etapa 3

Scheduled / always-on

Pasa a ejecución programada en runtime aislado. Sigue sin tocar gasto: analiza, alerta y prepara. Gates activos.

Etapa 4

Escritura con gates

Recién acá se habilitan escrituras (campañas, publicación), siempre detrás de approval gates y con presupuesto acotado.

→
Regla de diseño: la autonomía es una consecuencia del historial, no una feature que se activa el día uno. Un agente que pide permisos amplios desde el inicio es una mala configuración — y en 2026, con cuentas de ads y dominios en juego, también un riesgo innecesario.
